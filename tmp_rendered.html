Help

This page is meant to help remote fault finding on the VA-Connect encoder. The short descriptions below explain what each section means and how to use it.

Overview

Current diagnosis is the watchdog's best plain-English summary of what looks wrong right now.

Current state shows whether the watchdog currently thinks the box is healthy or in fault.

Unexpected reboots means the PC restarted without a recent watchdog reboot command, which can point to manual reboot, power issue, or hard crash.

Fault summary and Linux stability clues condense the logs into the simplest operator wording the watchdog can currently justify.

Current PC Stats

This row shows the most recent metric sample taken by the site watchdog.

CPU and Memory are percentages. MemAvailable is free RAM available to Linux. Cached is RAM being used for cache and can usually be reclaimed.

Temp max is the hottest thermal sensor currently exposed by Ubuntu. Load is the 1-minute load average.

Incident Timeline

This is the short event history for the latest watchdog activity.

Use it to see the sequence of startup, fault start, recovery, snapshots, reboot actions, and reboot detections.

PC Stats Chart

This chart trends the last 24 hours or 7 days of watchdog metrics.

CPU, memory, root disk, recording disk, and temperature are plotted together. Hover the graph to inspect a point in time.

The reboot markers show when the watchdog asked for a reboot or when a reboot was later detected. Numbers in brackets show how many of each marker are visible in the current chart range.

TeamViewer

This panel shows whether TeamViewer is installed, whether the daemon is up, and the detected TeamViewer ID.

Reset TeamViewer password asks the local TeamViewer CLI to generate a new password and shows it immediately so you can reconnect.

Start and Restart TeamViewer send daemon control commands for remote recovery when TeamViewer itself is the problem.

Latest Checks

App process is whether the watched VA-Connect process is running.

WAN checks prove internet reachability. TCP checks prove local devices such as the RUT or RTSP endpoint are reachable. Service checks prove selected systemd services are active.

Controls

Monitoring enabled turns automatic watchdog behaviour on or off.

App auto-restart allows the watchdog to start the app if it disappears.

Network restart before reboot tries the configured network restart command before a reboot. Reboot allowed controls whether the watchdog is allowed to reboot the PC.

Speed test runs a simple Cloudflare download/upload check from the codec and records Mbps along with CPU and memory usage at the time.

Investigation

Likely causes is a scoring guide based on the current evidence. It helps point you at memory/platform, storage, network, or app/service problems first.

Hardware warnings surfaces kernel, SMART, and persistent-crash clues. Crash review summarizes the latest previous-boot review if the box restarted.

Linux stability separates previous-boot clues from current-warning clues so you can judge whether NIC or memory messages were likely before the freeze or only after repower.

Config

App match is the process text the watchdog looks for. App start command is what it runs if that process disappears.

Internet hosts are public endpoints used to prove WAN access. TCP targets are local devices you want the watchdog to reach on host:port.

Base reboot timeout is the first wait before reboot. Max reboot timeout is the cap after backoff increases the reboot delay.

Logs And Files

The watchdog writes events to /var/log/va-connect-site-watchdog/events.jsonl and metrics to /var/log/va-connect-site-watchdog/metrics.jsonl.

Snapshots go under /var/log/va-connect-site-watchdog/snapshots. Current state lives in /var/lib/va-connect-site-watchdog/state.json.

Config

Use one internet host per line. Use one TCP target per line in the form 192.168.1.132:554.
